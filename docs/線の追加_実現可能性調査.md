# 線の追加 実現可能性調査

全画面編集ルートで管理したい線の種類と、現状のDB・画面構成での実現可否を調査した結果です。

---

## 0. 要件の整理（線種×色×太さの分解）

元要件を分解すると、以下の構造で表現可能：

| 軸 | 選択肢 |
|----|--------|
| **線の種類** | 単色 / 選択した色と白の縞線 / 選択した色の点線 |
| **線の色** | 赤 / 青 / 白 / グレー / ピンク |
| **太さ** | 選択（2,3,4,6 等） or サイズ入力 |

**現状の DB で対応可能か**: **可能**（スキーマ変更不要）

## 4. 実現可能性の判断

### 4.1 実現可能な部分

| 項目 | 判断 | 理由 |
|------|------|------|
| 線種のマスタ管理 | ✅ 可能 | html_object_value に value_code, value_name, value_data で管理可能 |
| 色・線種の指定 | ✅ 可能 | value_data に JSON やカンマ区切りで格納可能（例: `{"color":"#FF0000","style":"solid","width":2}`） |
| 道路の線・規制の線の複数種 | ✅ 可能 | html_object_value に複数レコード追加で対応 |
| design_data への保存 | ✅ 可能 | routes 配列に各線の座標・線種ID を格納可能 |

### 4.2 不足・対応が必要な部分

| 項目 | 不足内容 | 対応方法 |
|------|----------|----------|
| **html_object の区分** | 現状は「ルート描画」1区分のみ | 道路の線・規制の線を別区分にするか、ROUTE_DRAWING 内で value で管理するか設計が必要 |
| **value_data の拡張** | 色・線種・太さ・矢印の有無を格納する形式が未定義 | 構造化（JSON 等）で定義する必要あり |
| **CMYK の扱い** | 画面・地図では RGB/HEX に変換が必要 | value_data に CMYK と HEX の両方を格納するか、変換ロジックを用意 |
| **ルート1件の構造** | design_data.routes の各要素の型が unknown のまま | 座標・線種・太さ・矢印の有無を定義する型定義が必要 |
| **地図上の描画** | MapPreview は routes を描画していない | MapLibre の GeoJSON ソース＋レイヤーで線を描画する実装が必要 |
| **線の太さ・色の自由アレンジ** | UI が未実装 | カラーピッカー・太さスライダー等の UI 追加が必要 |
| **シードデータ** | html_object / html_object_value の初期データが未確認 | 要件に合わせた INSERT スクリプトが必要 |

## 5. 推奨対応方針

### 5.1 DB 設計

**案A: html_object を1つのカテゴリ「ルート描画」に集約**

- html_object: ROUTE_DRAWING 1件のみ
- html_object_value: 通行止、片側交互通行、迂回路、道路の線（白/グレー）、規制の線（赤/ピンク/青）、自由アレンジ を value_code で管理

**案B: 区分を分ける**

- html_object: ROUTE_DRAWING（通行止・片側交互・迂回路）、ROAD_LINE（道路の線）、RESTRICTION_LINE（規制の線）

→ 全画面編集では「線」として扱うため、**案A で value で管理**する方がシンプルで推奨。

### 5.2 value_data の拡張形式（推奨）

```json
{
  "color": "#FF0000",
  "cmyk": "0,100,100,0",
  "style": "solid",
  "width": 2,
  "arrow": false,
  "stripe": false
}
```

- 線種: solid / dashed / dotted
- 縞線: stripe: true のとき 2色を指定
- 矢印: arrow: true で迂回路の進行方向表示

### 5.3 design_data.routes の各要素の型（推奨）

```typescript
interface RouteItem {
  id: string
  type: string      // value_code または html_object_value_id
  coordinates: [number, number][]  // GeoJSON LineString
  width?: number
  color?: string    // 自由アレンジ時
}
```

### 5.4 実装タスク一覧

1. **DB**: html_object_value のシードデータ作成（通行止・片側交互・迂回路・道路の線・規制の線・自由アレンジ）
2. **設計**: value_data の JSON スキーマ定義、RouteItem 型定義
3. **フロント**: MapPreview に routes の描画ロジック追加（MapLibre GeoJSON LineString）
4. **フロント**: 全画面編集で「描画」ボタン押下時の地図上クリックで線を描画する UI
5. **フロント**: 自由アレンジ用のカラーピッカー・太さスライダー等の UI
6. **CMYK→RGB**: 変換ユーティリティ（印刷出力用に CMYK を保持する場合）

## 6. 結論

| 観点 | 結論 |
|------|------|
| **DB 構成** | 現状の html_object / html_object_value / design_data で十分対応可能。ただし value_data の形式とシードデータの整備が必要。 |
| **画面構成** | 全画面編集のサイドバーは既に html_object のルート描画を表示しているため、DB に追加すれば選択肢として表示可能。 |
| **不足点** | ① routes の描画ロジック（MapPreview 実装）、② 地図上での線描画 UI、③ 自由アレンジの UI、④ value_data の JSON 形式定義、⑤ シードデータの投入。 |

**総合判断**: 現状の DB・画面構成で**実現可能**。上記の不足点を順次対応すれば、要件の線種を全画面編集ルートで管理できる。

## 7. 線種×色×太さの分解構造での対応方法

### 7.1 方式A: 組み合わせを事前登録（1つのドロップダウン）【推奨】

| 項目 | 内容 |
|------|------|
| **DB 変更** | 不要 |
| **html_object** | ROUTE_DRAWING 1件のまま |
| **html_object_value** | 線種3 × 色5 × 太さ4 = 60件 を登録（またはよく使う組み合わせのみ 15〜20件） |
| **value_data** | `{"color":"#HEX","style":"solid または dashed","stripe":bool,"width":N}` で表現 |
| **UI** | 1つのドロップダウンで「通行止・車両通行止」「片側交互通行」等の表示名を選択 |

**value_data の例:**
- 単色・赤・4px: `{"color":"#E60012","style":"solid","stripe":false,"width":4}`
- 縞線・青白・4px: `{"color":"#0066B3","color2":"#FFFFFF","style":"solid","stripe":true,"width":4}`
- 点線・青・4px: `{"color":"#0033A0","style":"dashed","stripe":false,"width":4}`

**太さの自由入力**: 例として `CUSTOM` の value を1件用意し、その選択時は太さ入力欄を表示。value_data に width を含めず、UI で入力した値を使用。

### 7.2 方式B: 線種・色・太さを別々に選択（3つのドロップダウン）

| 項目 | 内容 |
|------|------|
| **DB 変更** | スキーマ変更不要。html_object を 3 件追加 |
| **html_object 追加** | ROUTE_LINE_TYPE（単色/縞線/点線）, ROUTE_LINE_COLOR（赤/青/白/グレー/ピンク）, ROUTE_LINE_WIDTH（2,3,4,6 等） |
| **html_object_value** | 各 html_object に選択肢を登録 |
| **UI 変更** | OrderDetail の「ルート描画」セクションで複数ドロップダウンを表示するロジックが必要 |

**注意**: 方式B の場合、OrderDetail の UI を「ルート描画」セクション内に 3 つのドロップダウンを出すように変更する必要がある。現状は html_object 1件 = 1 ドロップダウンのループ表示になっているため、グループ化や表示制御のロジック追加が必要。

### 7.3 結論

| 方式 | 現状 DB で対応 | 推奨 |
|------|----------------|------|
| 方式A（組み合わせ事前登録） | ✅ 可能 | ◎ UI 変更最小 |
| 方式B（3つのドロップダウン） | ✅ 可能 | △ UI 変更が必要 |
