# トラブルシューティング

よくある事象と対処方法。

---

## 1. DBeaver で RDS に接続できない

### 症状

- `nc -zv signage-dev-db.xxx.rds.amazonaws.com 3306` がタイムアウト
- DBeaver で接続エラー

### 原因と対処

| 原因 | 対処 |
|------|------|
| **IP が変わった** | `./scripts/update-db-sg-my-ip.sh` を実行。または EC2 → セキュリティグループ → signage-db-sg で「マイ IP」を再保存 |
| **ルートテーブル** | RDS のサブネットのルートが IGW であること。NAT だとインターネットからの接続不可 |
| **パブリックアクセス** | RDS の「パブリックアクセス」が「あり」であること |

### 接続テスト

```bash
nc -zv signage-dev-db.cly840668ik8.ap-northeast-1.rds.amazonaws.com 3306
```

## 2. 住所検証・地図中心表示が動かない

### 症状

- 枝番エリアで「Failed to fetch」
- 住所検証がタイムアウト
- 地図の中心が東京駅のまま

### 原因

Lambda が MapTiler API に到達できていない。Lambda は **NAT 経由のサブネット** に配置する必要がある。

### 対処

1. Lambda の VPC 設定を確認
2. サブネットが NAT ルートテーブルに紐づいていること
3. 現行: subnet-0ce8f2d34f7d880cd, subnet-07ab8df265549f7fe

### 確認

```bash
# 住所検証
curl "https://tpdeco04lg.execute-api.ap-northeast-1.amazonaws.com/prod/address/validate?address=東京駅"

# ジオコーディング
curl "https://tpdeco04lg.execute-api.ap-northeast-1.amazonaws.com/prod/address/geocode?address=東京駅"
```

## 3. Lambda が RDS に接続できない（500 エラー）

### 症状

- `/health` は成功するが `/companies` 等が 500
- CloudWatch: `Lost connection to MySQL server during query`

### 原因と対処

| 原因 | 対処 |
|------|------|
| **RDS の SG に RDS Proxy が未許可** | signage-db-sg のインバウンドに sg-09651f78bc62d327c からの 3306 を追加 |
| **DATABASE_URL が RDS 直** | Lambda の DATABASE_URL を RDS Proxy のエンドポイントに変更 |
| **RDS Proxy ターゲットが UNAVAILABLE** | 上記 SG 設定後、数分待つ |

### 確認

```bash
# RDS Proxy ターゲット状態
aws rds describe-db-proxy-targets --db-proxy-name signage-dev-proxy --region ap-northeast-1
```

---

## 4. DBeaver と地図機能の両立

RDS と Lambda を別サブネットに分離する必要がある:

| 用途 | サブネット | ルート |
|------|------------|--------|
| RDS（DBeaver） | subnet-08acbf1c74b943f8f | IGW |
| Lambda（MapTiler） | subnet-0ce8f2d34f7d880cd, subnet-07ab8df265549f7fe | NAT |

詳細は [02.VPC・ネットワーク構築手順.md](02.VPC・ネットワーク構築手順.md) 参照。
