# Lambda と RDS の接続トラブルシューティング

Lambda から RDS への接続がタイムアウトする場合の、原因特定と修正手順。

---

## 調査事例（2026-03-09）

### 事象

- `/companies` が 500 エラー
- CloudWatch: `(2013, 'Lost connection to MySQL server during query')`

### 根本原因

**RDS インスタンスの SG に、RDS Proxy の SG からの 3306 が許可されていなかった。**

接続経路は `Lambda → RDS Proxy → RDS`。Lambda は RDS Proxy に届いていたが、RDS Proxy が RDS に接続できず、ハンドシェイク中に切断されていた。

### 確認したこと

| 項目 | 結果 |
|------|------|
| `describe-db-proxy-targets` | 当初 `UNAVAILABLE`（internal error） |
| RDS の SG インバウンド | RDS Proxy の SG（sg-09651f78bc62d327c）が**未許可** |
| RDS Proxy の SG インバウンド | Lambda の SG からの 3306 は許可済み ✓ |

### 実施した修正

1. **RDS の SG**（sg-04ebd02d056405e61）に、**RDS Proxy の SG**（sg-09651f78bc62d327c）からの 3306 を許可するルールを追加
2. 数分後、RDS Proxy ターゲットが `AVAILABLE` に変化
3. RDS Proxy 経由で API が正常動作することを確認

### AWS CLI での SG ルール追加

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-04ebd02d056405e61 \
  --protocol tcp --port 3306 \
  --source-group sg-09651f78bc62d327c \
  --region ap-northeast-1
```

### 補足: RDS Proxy が使えない場合の代替

RDS Proxy のターゲットが UNAVAILABLE の間、Lambda の DATABASE_URL を **RDS インスタンス直**（`signage-dev-db.xxx.rds.amazonaws.com`）に変更することで接続可能。Lambda と RDS が同じ SG の場合、直接接続は許可されている。

---

## 症状

- `/health` は成功する（`{"status":"ok"}`）
- `/companies` や `/users/by-auth-uid` は 500 エラー
- CloudWatch ログに以下のいずれかが表示される:
  - `TimeoutError: timed out`（接続が届かない）
  - `pymysql.err.OperationalError: (2013, 'Lost connection to MySQL server during query')`（接続は届くがハンドシェイク中に切断）

---

## 確認手順（順番に実施）

### 手順 1: Lambda の DATABASE_URL を確認

**目的**: Lambda が RDS Proxy に接続しているか確認する。

1. AWS コンソールにログイン
2. 検索バーで「**Lambda**」と入力 → Lambda を開く
3. 関数一覧から **`signage-dev-api`** をクリック
4. **設定** タブをクリック
5. 左メニュー **「環境変数」** をクリック
6. **DATABASE_URL** の値を確認

**正しい形式の例**:
```
mysql+pymysql://admin:パスワード@signage-dev-proxy.proxy-xxxxx.ap-northeast-1.rds.amazonaws.com:3306/signage_dev?charset=utf8mb4
```

**確認ポイント**:
- ホスト名に **`proxy`** が含まれているか
  - ✅ 正しい: `signage-dev-proxy.proxy-cly840668ik8.ap-northeast-1.rds.amazonaws.com`
  - ❌ 誤り: `signage-dev-db.cly840668ik8.ap-northeast-1.rds.amazonaws.com`（`-db` は RDS インスタンス）

**修正方法**（誤っている場合）:
1. 「編集」をクリック
2. DATABASE_URL の値を修正
3. ホスト名を RDS Proxy のエンドポイントに変更
4. RDS Proxy のエンドポイントは: RDS コンソール → プロキシ → `signage-dev-proxy` → エンドポイントをコピー
5. 「保存」をクリック

---

### 手順 2: RDS Proxy のターゲット状態を確認

**目的**: RDS Proxy が RDS に接続できているか確認する。

```bash
aws rds describe-db-proxy-targets --db-proxy-name signage-dev-proxy --region ap-northeast-1
```

- `TargetHealth.State` が **AVAILABLE** であること
- **UNAVAILABLE** の場合、`Description` を確認。多くは RDS の SG で RDS Proxy の SG が許可されていないことが原因

### 手順 3: RDS Proxy のエンドポイントを取得

**目的**: 正しい接続先を確認する。

1. 検索バーで「**RDS**」と入力 → RDS を開く
2. 左メニュー **「プロキシ」** をクリック
3. **`signage-dev-proxy`** をクリック
4. 「接続とセキュリティ」または詳細画面で **エンドポイント** を確認
   - 例: `signage-dev-proxy.proxy-cly840668ik8.ap-northeast-1.rds.amazonaws.com`

このエンドポイントが DATABASE_URL のホスト部分と一致しているか確認する。

---

### 手順 4: Lambda の VPC 設定を確認

**目的**: Lambda が RDS Proxy と同じネットワークにいるか確認する。

1. Lambda コンソール → `signage-dev-api` → **設定** タブ
2. 左メニュー **「VPC」** をクリック
3. 以下を確認:
   - **VPC**: 何か選択されているか（未設定だと RDS に接続できない）
   - **サブネット**: 2 つ以上選択されているか
   - **セキュリティグループ**: `signage-db-sg` または `signage-proxy-sg` が選択されているか

**未設定の場合**: Lambda は VPC 外にあり、RDS Proxy（プライベートサブネット）に接続できない。VPC を設定する必要がある。

---

### 手順 5: セキュリティグループのインバウンドルールを確認

**目的**: RDS Proxy が Lambda からの接続を許可しているか確認する。

1. 検索バーで「**EC2**」と入力 → EC2 を開く
2. 左メニュー **「セキュリティグループ」** をクリック
3. RDS Proxy にアタッチされている SG を探す:
   - 名前で検索: `signage-db-sg` または `signage-proxy-sg`
   - または RDS コンソール → プロキシ → `signage-dev-proxy` → セキュリティグループを確認
4. 該当 SG をクリック → **インバウンドルール** タブを開く
5. 以下のルールが存在するか確認:

| タイプ | ポート | ソース |
|--------|--------|--------|
| MySQL/Aurora | 3306 | 同じ SG の ID（例: sg-xxxxxxxx） |

**ソース** は Lambda に割り当てている SG の ID である必要がある。Lambda と RDS Proxy が同じ SG を使っている場合は、その SG 自身の ID を指定する（自己参照ルール）。

**ルールが無い場合**:
1. 「インバウンドルールを編集」をクリック
2. 「ルールを追加」をクリック
3. タイプ: **MySQL/Aurora**、ポート: **3306**
4. ソース: **カスタム** を選択し、Lambda の SG ID を入力（Lambda と RDS Proxy が同じ SG なら、その SG の ID）
5. 「ルールを保存」をクリック

---

### 手順 6: Lambda と RDS Proxy が同じサブネットにいるか確認

**目的**: ネットワーク経路が通っているか確認する。

1. Lambda の VPC 設定で表示されている **サブネット ID** をメモ（例: subnet-aaa, subnet-bbb）
2. RDS コンソール → プロキシ → `signage-dev-proxy` → **サブネット** を確認
3. **同じ VPC 内** のサブネットか確認（必ずしも同じサブネットである必要はないが、同じ VPC である必要がある）
4. Lambda の VPC と RDS Proxy の VPC が一致しているか確認

---

### 手順 7: 修正後の動作確認

上記の修正を行った後:

1. Lambda の環境変数を変更した場合、**次回の呼び出しで自動的に反映**される（再デプロイ不要）
2. セキュリティグループを変更した場合、**即時反映**される
3. 以下で動作確認:

```bash
curl "https://tpdeco04lg.execute-api.ap-northeast-1.amazonaws.com/prod/companies"
```

成功時は JSON 形式で会社一覧が返る。

---

### 手順 8: エラー 2013「Lost connection」が出る場合（SSL / RDS Proxy → RDS）

**症状**: `(2013, 'Lost connection to MySQL server during query')` が CloudWatch に表示される。

**原因候補**:
1. **RDS Proxy が TLS を要求**しているが、アプリが SSL 未使用で接続している
2. **RDS Proxy → RDS** の接続が許可されていない（RDS の SG 設定）

**対応 1: アプリ側の SSL 有効化**

バックエンドの `app/db/session.py` で、`rds.amazonaws.com` を含む DATABASE_URL の場合は自動的に **mysql-connector-python** に切り替え、SSL（certifi の CA バンドル）で接続する。Lambda を再デプロイして反映させる。

**対応 2: RDS の SG で RDS Proxy を許可**

RDS Proxy が RDS インスタンスに接続するため、**RDS インスタンスにアタッチされている SG** のインバウンドに、**RDS Proxy の SG** からの 3306 を許可するルールが必要。

1. EC2 コンソール → セキュリティグループ
2. **RDS インスタンス**にアタッチされている SG（例: `signage-db-sg`）を開く
3. インバウンドルールに以下を追加:
   - タイプ: **MySQL/Aurora**、ポート: **3306**
   - ソース: **RDS Proxy の SG ID**（例: `sg-09651f78bc62d327c`）
4. RDS Proxy の SG は RDS コンソール → プロキシ → `signage-dev-proxy` → セキュリティグループで確認
