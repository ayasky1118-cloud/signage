# 構築手順クイックリファレンス

実施済みの内容に基づく、正しい構築・運用のチェックリスト。

## 初回構築の順序

1. **環境準備**（00）: AWS CLI、Docker、mysql クライアント
2. **RDS 作成**（02-RDS）: signage-dev-db、signage-db-sg、マイ IP で 3306 許可
3. **Secrets Manager**（02 Phase 3）: signage-dev/db-credentials
4. **RDS Proxy**（02 Phase 4）: signage-dev-proxy
5. **【必須】RDS の SG に RDS Proxy の SG を許可**（02 Phase 4.2）
   - 忘れると RDS Proxy ターゲットが UNAVAILABLE、Lambda が 500 エラー
6. **Lambda**（02 Phase 5）: VPC、SG、環境変数
7. **API Gateway**（02 Phase 6）: HTTP API、CORS
8. **スキーマ投入**: RDS に db/00_schema/ 001〜012 を実行
9. **データ移行**（06）: migrate-to-rds.sh でローカル → RDS
10. **auth_uid 設定**: user テーブルの auth_uid を Cognito sub に合わせる

## Lambda デプロイ（コード変更時）

```bash
cd signage/backend
rm -rf package && mkdir package
pip3 install -r requirements.txt -t package/ \
  --platform manylinux2014_x86_64 --python-version 3.12 --only-binary=:all:
cp -r app package/ && cp handler.py package/
cd package && zip -r ../lambda-deploy.zip . -x "*.pyc" -x "*__pycache__*" && cd ..
aws lambda update-function-code \
  --function-name signage-dev-api \
  --zip-file fileb://lambda-deploy.zip \
  --region ap-northeast-1 --no-cli-pager
```

## セキュリティグループの必須ルール

| 接続元 | 接続先 | ルール |
|--------|--------|--------|
| Lambda | RDS Proxy | RDS Proxy の SG で Lambda の SG からの 3306 を許可 |
| RDS Proxy | RDS | **RDS の SG で RDS Proxy の SG からの 3306 を許可**（必須） |
| ローカル | RDS | RDS の SG でマイ IP からの 3306 を許可（データ移行用） |

※ IP が変わって接続できなくなった場合: `./scripts/update-db-sg-my-ip.sh`

## データ移行・復元

| 方向 | スクリプト | 用途 |
|------|------------|------|
| ローカル → RDS | `migrate-to-rds.sh` | 本番反映、初回投入 |
| RDS → ローカル | `migrate-from-rds.sh` | ローカル DB 復元 |

## ネットワーク構成（DBeaver + 地図機能の両立）

| 用途 | サブネット | ルート | 備考 |
|------|------------|--------|------|
| RDS（DBeaver 接続） | subnet-08acbf1c74b943f8f 等 | 0.0.0.0/0 → IGW | インターネットからの接続を許可 |
| Lambda（MapTiler API） | subnet-0ce8f2d34f7d880cd, subnet-07ab8df265549f7fe | 0.0.0.0/0 → NAT | 外向きのみ、住所検証・ジオコーディング用 |

※ RDS と Lambda を別サブネットに分離することで、DBeaver 接続と地図機能を同時に利用可能。
