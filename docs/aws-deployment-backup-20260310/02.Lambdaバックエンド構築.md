# 02. Lambda バックエンド構築

バックエンドを **API Gateway + Lambda** で構築する。  
フロントエンドは Amplify で既に設定済みを前提とする。

---

## 構築するリソース一覧

| リソース | 用途 |
|----------|------|
| VPC | ネットワーク基盤 |
| Subnet | プライベート（RDS・Lambda）・パブリック（NAT 等） |
| RDS for MySQL | データベース |
| RDS Proxy | Lambda からの DB 接続プーリング |
| Secrets Manager | DB 認証情報の管理 |
| Lambda | FastAPI（Mangum）の実行 |
| API Gateway | HTTP API のエンドポイント |
| Cognito | 認証（既存 or 新規） |

---

## アーキテクチャ概要

```
[Amplify フロント] ──→ [API Gateway] ──→ [Lambda]
                                              │
                                              ↓
                                        [RDS Proxy]
                                              │
                                              ↓
                                        [RDS MySQL]
```

---

## 前提

- フロントエンド（Amplify）は設定済み
- AWS CLI がインストール・設定済み（[00.環境準備手順.md](00.環境準備手順.md) 参照）
- リージョン: **ap-northeast-1（東京）**

---

## Phase 1: VPC と Subnet

### 1.1 デフォルト VPC を使用する場合

そのまま利用可能。サブネット ID を控える。

```bash
aws ec2 describe-subnets --filters "Name=default-for-az,Values=true" \
  --query "Subnets[*].[SubnetId,AvailabilityZone,CidrBlock]" \
  --output table --region ap-northeast-1
```

### 1.2 カスタム VPC を作成する場合

1. **VPC** → **VPC の作成**
2. 名前: `signage-dev-vpc`
3. IPv4 CIDR: `10.0.0.0/16`
4. パブリックサブネット 2 つ、プライベートサブネット 2 つ（推奨）
5. NAT ゲートウェイ: プライベートサブネットからインターネットアクセス用

※ Lambda を VPC 内に配置する場合、RDS への接続のためプライベートサブネットに配置する。その場合 NAT ゲートウェイ（または VPC エンドポイント）が必要。

## Phase 2: RDS for MySQL

コンソールでの詳細手順は [02.RDS作成手順（コンソール）.md](02.RDS作成手順（コンソール）.md) を参照。

### 2.1 作成手順（概要）

1. **RDS** → **データベースの作成**
2. **エンジン**: MySQL 8.0、**テンプレート**: 開発/テスト
3. **デプロイオプション**: シングル AZ（1 インスタンス）
4. **設定**: DB 識別子 `signage-dev-db`、マスターユーザー `admin`、パスワードを控える
5. **インスタンス**: db.t3.micro
6. **ストレージ**: 20 GB（gp3）
7. **接続**: デフォルト VPC、パブリックアクセス「あり」、SG 新規 `signage-db-sg`
8. **認証**: セルフマネージド、パスワード認証
9. **RDS Proxy**: チェック OFF（Phase 4 で別途作成）
10. **作成** → ステータス「利用可能」まで待つ

### 2.2 セキュリティグループ

`signage-db-sg` のインバウンド（RDS 作成直後は「マイ IP」のみのことが多い）:
- MySQL/Aurora (3306): ローカル用に「マイ IP」を追加（[02.RDS作成手順（コンソール）](02.RDS作成手順（コンソール）.md) の「16. セキュリティグループ」参照）
- **Lambda 用**: Phase 5 の **5.0 事前準備** で、signage-db-sg 自身からの 3306 を追加する

## Phase 3: Secrets Manager

### 3.1 DB 認証情報の登録

1. **Secrets Manager** → **新しいシークレットの保存**
2. **シークレットのタイプ**: その他のシークレットのタイプ
3. **キー/値**:
   - `username`: RDS のマスターユーザー名
   - `password`: RDS のマスターパスワード
4. **シークレット名**: `signage-dev/db-credentials`
5. **作成**

※ RDS Proxy は Secrets Manager のシークレットを参照する。

## Phase 4: RDS Proxy

### 4.1 作成手順

1. **RDS** → **プロキシ** → **プロキシの作成**
2. **プロキシ識別子**: `signage-dev-proxy`
3. **エンジン**: MySQL
4. **認証**: Secrets Manager のシークレット `signage-dev/db-credentials` を選択
5. **サブネット**: RDS と同じ VPC のプライベートサブネット
6. **セキュリティグループ**: 新規 `signage-proxy-sg` または `signage-db-sg`（既存を流用する場合は signage-db-sg）
7. **ターゲットグループ**: RDS インスタンス `signage-dev-db` を選択
8. **作成**

**RDS Proxy を signage-proxy-sg で作成した場合**、以下の SG 設定が必要:
- `signage-db-sg`: インバウンドに signage-proxy-sg からの 3306 を追加（RDS Proxy → RDS）
- `signage-proxy-sg`: インバウンドに signage-proxy-sg 自身からの 3306 を追加（Lambda → RDS Proxy。Phase 5.0 で実施）

**signage-db-sg を流用した場合**は、Phase 5.0 で signage-db-sg の自己参照ルールを追加すればよい。

### 4.2 接続文字列

RDS Proxy のエンドポイントを控える。形式:
```
signage-dev-proxy.xxxxx.ap-northeast-1.rds.amazonaws.com:3306
```

**DATABASE_URL**（後で Lambda に渡す）:
```
mysql+pymysql://admin:パスワード@signage-dev-proxy.xxxxx.ap-northeast-1.rds.amazonaws.com:3306/signage_dev?charset=utf8mb4
```

※ パスワードは Secrets Manager に格納済み。Lambda の環境変数では Secrets Manager の ARN を参照する方法もある（後述）。

## Phase 5: Lambda 関数の作成

### 5.0 事前準備: セキュリティグループ（Lambda 作成前に必須）

Lambda が RDS Proxy に接続するには、**Lambda 作成前に** RDS Proxy のセキュリティグループで Lambda からの 3306 を許可する必要がある。

**Lambda に `signage-db-sg` を割り当てる場合**（RDS Proxy が signage-db-sg を使っている場合）:

1. **EC2** → **セキュリティグループ** → `signage-db-sg` を選択
2. **インバウンドルールを編集** → **ルールを追加**
3. 以下を設定:
   - **タイプ**: MySQL/Aurora
   - **ポート**: 3306
   - **ソース**: **カスタム** を選択し、`sg-xxxxxxxx`（signage-db-sg 自身の ID）を入力。「マイ IP」や IP アドレスは選ばない
4. **ルールを保存**

※ 「マイ IP」のみの状態だと Lambda から接続できない。上記ルールを追加してから Lambda を作成すること。ソースを IP にすると重複エラーになる場合がある。

**Lambda に `signage-proxy-sg` を割り当てる場合**（Phase 4 で RDS Proxy 用に新規作成した場合）:

- `signage-proxy-sg` に同様に「自分自身からの 3306」を許可するルールを追加する。

---

### 5.1 デプロイパッケージの準備

Lambda 用に ZIP を作成する。依存関係を含める必要がある。

**重要**: Lambda は Linux で動作する。macOS でビルドする場合、`--platform manylinux2014_x86_64` を指定しないと pydantic_core 等のネイティブ拡張が動かず `ImportModuleError` になる。

```bash
cd signage/backend
rm -rf package && mkdir package
# macOS からビルドする場合: Linux 用パッケージを取得
pip3 install -r requirements.txt -t package/ \
  --platform manylinux2014_x86_64 \
  --python-version 3.12
cp -r app package/
cp handler.py package/
cd package
# .pyc と __pycache__ のみ除外。*.so は除外しない（pydantic_core 等に必要）
zip -r ../lambda-deploy.zip . -x "*.pyc" -x "*__pycache__*"
cd ..
```

※ Linux 環境でビルドする場合は `--platform` オプションは不要。  
※ ZIP が 50MB 超でアップロードがタイムアウトする場合は、S3 経由でアップロードする。

---

### 5.2 Lambda 関数の作成

1. **Lambda** → **関数の作成**
2. **関数名**: `signage-dev-api`
3. **ランタイム**: Python 3.12
4. **アーキテクチャ**: x86_64（または arm64）
5. **詳細設定**:
   - VPC: RDS Proxy と同じ VPC
   - サブネット: **2 つ以上**を選択（異なる AZ 推奨）。デフォルト VPC の場合は表示されるサブネットを複数選択
   - セキュリティグループ: **signage-db-sg** または **signage-proxy-sg**（RDS Proxy にアタッチしている SG と同じもの）
6. **作成**

※ 5.0 で SG のインバウンドを設定していないと、作成後も Lambda が RDS Proxy に接続できない。

※ **地図表示（ジオコーディング）** を使う場合、Lambda は MapTiler API（`api.maptiler.com`）を呼び出すため **インターネットアクセス** が必要。プライベートサブネットの場合は **NAT ゲートウェイ** を設置すること（Phase 1 参照）。未設定だと枝番エリアで「Failed to fetch」となる。

### 5.3 ハンドラー・タイムアウト・メモリの設定

- **ハンドラー**: `handler.handler`
- **タイムアウト**: 30 秒
- **メモリ**: 512 MB（必要に応じて調整）

**コンソールでの設定場所**（UI により異なる）:
- **タイムアウト・メモリ**: 設定 → 一般設定 → 編集
- **ハンドラー**: コンソールによっては「ランタイムの設定」が別メニューにない場合がある

**AWS CLI で設定する方法**（確実）:

```bash
aws lambda update-function-configuration \
  --function-name signage-dev-api \
  --handler handler.handler \
  --runtime python3.12 \
  --region ap-northeast-1 \
  --no-cli-pager
```

タイムアウト・メモリは「設定」→「一般設定」→「編集」で変更。`--no-cli-pager` は出力がページャで止まるのを防ぐ。

### 5.4 環境変数

| キー | 値 |
|------|-----|
| `DATABASE_URL` | `mysql+pymysql://admin:パスワード@RDS_PROXY_ENDPOINT:3306/signage_dev?charset=utf8mb4` |
| `API_PREFIX` | `/prod`（API Gateway の $default ステージでは URL にステージ名を含めない場合も、アプリ側のパスとして `/prod` を使用） |
| `CORS_ORIGINS` | Amplify の URL（例: `https://main.xxxxx.amplifyapp.com`）。Amplify コンソール → アプリ → ブランチの「ドメイン」で確認 |
| `MAPTILER_API_KEY` | （任意）MapTiler の API キー。枝番エリアの地図表示に必要。未設定時は 503 エラー |

**AWS CLI で一括設定**:

```bash
aws lambda update-function-configuration \
  --function-name signage-dev-api \
  --environment "Variables={DATABASE_URL=mysql+pymysql://admin:パスワード@RDS_PROXY_ENDPOINT:3306/signage_dev?charset=utf8mb4,API_PREFIX=/prod,CORS_ORIGINS=https://main.xxxxx.amplifyapp.com,MAPTILER_API_KEY=MapTilerで発行したキー}" \
  --region ap-northeast-1 \
  --no-cli-pager
```

※ 本番では `DATABASE_URL` を Secrets Manager から参照することを推奨。

### 5.5 IAM ロール

Lambda 実行ロールに以下を付与:
- `AWSLambdaVPCAccessExecutionRole`（VPC アクセス）
- RDS Proxy への接続のため、VPC 内のネットワークアクセスを許可
- （Secrets Manager 使用時）`SecretsManagerReadWrite` または該当シークレットへのアクセス

## Phase 6: API Gateway

### 6.1 HTTP API の作成

1. **API Gateway** → **API の作成** → **HTTP API** → **構築**
2. **ステップ 1: API を設定**
   - API 名: `signage-dev-api`
   - **統合**: 統合タイプ「Lambda」、AWS リージョン「ap-northeast-1」、**Lambda 関数**の欄に `signage-dev-api` を入力または選択
3. **次へ**
4. **ステップ 2: ルートを設定**
   - メソッド `ANY`、リソースパス `/{proxy+}`、統合 `signage-dev-api`
   - **ルートを追加** で、メソッド `ANY`、リソースパス `/`、統合 `signage-dev-api`
5. **次へ**
6. **ステップ 3: ステージ** … `$default` のまま（自動デプロイ有効）
7. **次へ** → **作成**

### 6.2 ルートの確認

- `ANY /{proxy+}` と `ANY /` の 2 つが Lambda に統合されていることを確認

### 6.3 デプロイと URL の確認

作成後、**URL の呼び出し**が表示されない場合:

1. 左メニュー **「ステージ」** をクリック
2. **「$default」** を選択
3. **「URL の呼び出し」** をコピー（例: `https://xxxxx.execute-api.ap-northeast-1.amazonaws.com`）

手動デプロイが必要な場合（「デプロイ」ボタンが表示される場合）:

1. 画面上部の **「デプロイ」** をクリック
2. ステージで **「$default」** を選択
3. **「デプロイ」** を実行

### 6.4 CORS（API Gateway 側）

左メニュー **「CORS」** で、Amplify のオリジン（例: `https://main.xxxxx.amplifyapp.com`）を許可する。

## トラブルシューティング

| 現象 | 確認事項 |
|------|----------|
| `{"message":"Internal Server Error"}` | CloudWatch Logs でエラー内容を確認。Lambda → モニタリング → CloudWatch のログを表示 |
| `ImportModuleError: No module named 'pydantic_core._pydantic_core'` | macOS でビルドしたパッケージが原因。5.1 の `--platform manylinux2014_x86_64` で再ビルド。ZIP 作成時に `*.so` を除外していないか確認 |
| Connection was closed before response | ZIP が大きい、またはネットワークタイムアウト。`*.pyc` `*__pycache__*` を除外して軽量化。50MB 超の場合は S3 経由でアップロード |
| Lambda が RDS に接続できない（TimeoutError） | [07.Lambda-RDS接続トラブルシューティング.md](07.Lambda-RDS接続トラブルシューティング.md) の手順で原因を特定 |
| CORS エラー | CORS_ORIGINS に Amplify の URL が正しく設定されているか。API Gateway の CORS 設定も確認 |
| 地図読み込み「Failed to fetch」 | [地図読み込み失敗のトラブルシューティング](#地図読み込み失敗failed-to-fetch) を参照 |
| 502 Bad Gateway | Lambda のログ（CloudWatch）を確認。ハンドラー名 `handler.handler` が正しいか |
| コールドスタートが遅い | メモリを増やす、Provisioned Concurrency の検討 |

## 地図読み込み失敗（Failed to fetch）

枝番エリアで「地図を読み込み中...」のまま「Failed to fetch」となる場合の確認事項。全画面でも注文住所を地図の中心に表示するには同じジオコーディング API が必要で、失敗時は地図は表示されるが中心が東京駅のままとなる。

### 原因 1: Lambda がインターネットに接続できない（MapTiler API 呼び出し不可）

Lambda はジオコーディングで **MapTiler API**（`api.maptiler.com`）を呼び出す。Lambda が **プライベートサブネット** にあり、**NAT ゲートウェイがない**場合、外部 API に到達できずタイムアウトする。

**確認**:
1. Lambda → 設定 → VPC → サブネットを確認
2. そのサブネットのルートテーブルに `0.0.0.0/0` が **NAT ゲートウェイ** 経由になっているか確認
3. デフォルト VPC のパブリックサブネットの場合は、インターネットゲートウェイ経由で接続可能

**対応**: プライベートサブネットを使用している場合、**NAT ゲートウェイ** を設置する（Phase 1 参照）。または、Lambda をパブリックサブネットに配置し、パブリック IP を有効にする（VPC の設計により選択）。

### 原因 2: CORS 設定

**確認**:
1. Lambda の環境変数 `CORS_ORIGINS` に Amplify の URL が**完全一致**で設定されているか（例: `https://main.d27nzen1vhtcee.amplifyapp.com`）
2. API Gateway → CORS で、上記オリジンが許可されているか

### 原因 3: MAPTILER_API_KEY 未設定

Lambda の環境変数に `MAPTILER_API_KEY` が設定されているか確認。未設定時は 503 エラーとなり、「地図表示の設定がありません」と表示される（Failed to fetch とは異なる）。

### 動作確認（curl）

ブラウザ以外から API が応答するか確認する:

```bash
# API Gateway の URL に置換。address は「東京都千代田区神田神保町」の URL エンコード
curl "https://xxxxx.execute-api.ap-northeast-1.amazonaws.com/prod/address/geocode?address=%E6%9D%B1%E4%BA%AC%E9%83%BD%E5%8D%83%E4%BB%A3%E7%94%B0%E5%8C%BA%E7%A5%9E%E7%94%B0%E7%A5%9E%E4%BF%9D%E7%94%BA"
```

- **成功時**: `{"lng":139.75...,"lat":35.69...}` が返る
- **503**: MAPTILER_API_KEY 未設定
- **タイムアウト**: Lambda が MapTiler に到達できていない（NAT 等を確認）
