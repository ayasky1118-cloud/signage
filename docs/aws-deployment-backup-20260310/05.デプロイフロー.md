# デプロイフロー

検証環境（Amplify main URL）へのデプロイ手順と、古いビルドで上書きされないための注意点。

※ 現在構築済みの AWS（Amplify + Lambda + RDS）は**検証環境**。本番は将来構築予定。

---

## デプロイの仕組み（要確認）

main へのデプロイが **複数の経路** から行われていると、後から完了した方が上書きします。

| 経路 | 説明 | 確認場所 |
|------|------|----------|
| **GitLab CI** | `.gitlab-ci.yml` の build → deploy（zip を Amplify にアップロード） | パイプライン一覧 |
| **Amplify Git 連携** | Amplify がリポジトリに接続され、push で自動ビルド | Amplify コンソール「ホスティング」 |
| **GitLab Auto DevOps** | プロジェクトで Auto DevOps が有効な場合、追加のデプロイが動く可能性 | Settings → CI/CD → Auto DevOps |

**推奨**: デプロイ経路は **1つに統一**する。本プロジェクトでは GitLab CI（`.gitlab-ci.yml`）でデプロイしている想定。Amplify の Git 連携や Auto DevOps が有効なら、どちらかを無効化するか、役割を明確に分ける。

---

## 正しい手順

```
dev で開発 → main にマージ → main を push → CI がビルド・デプロイ
```

| ステップ | 作業 |
|----------|------|
| 1 | `dev` ブランチで開発・コミット |
| 2 | `main` にマージ（`git checkout main && git merge dev`） |
| 3 | `git push origin main` および `git push signage_dev main` |
| 4 | GitLab の main でパイプラインが実行され、Amplify にデプロイ |

---

## 重要なルール

### 検証デプロイは main のみ

- **Amplify の main URL**（`main.d27nzen1vhtcee.amplifyapp.com`）には、**main ブランチの push でしかデプロイしない**
- `dev` ブランチのパイプラインが検証 URL にデプロイされないよう、CI は `only: refs: main` に限定している

### 古いパイプラインによる上書きを防ぐ

- マージ前に、**古い main 用のパイプラインが実行中・保留中で残っていないか**確認する
- 古いパイプラインが後から完了すると、新しいデプロイを古いビルドで上書きする
- マージ・push の前に、該当パイプラインをキャンセルするか、完了を待ってから進める

### 推奨タイミング

1. 古い main のパイプラインをキャンセルする（あれば）
2. `dev` を `main` にマージ
3. `main` を push
4. 新しいパイプラインの完了を待つ

## トラブル時の確認

| 現象 | 確認すること |
|------|--------------|
| 画面が古いまま | ブラウザの強制リロード（Cmd+Shift+R）、最新パイプラインの成功有無 |
| 古いビルドで上書きされた | パイプライン一覧で、古い main のパイプラインが後から完了していないか確認 |
| パイプラインが保留中 | Runner のタグ一致、Runner の稼働状況（[04.GitLab Runner設定.md](04.GitLab Runner設定.md)） |
